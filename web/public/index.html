这个页面现在更像一个执行控制台。你不需要先理解所有 JSON 和事件模型，只要按推荐路径走，就能在几分钟内看懂 task、workflow、流式输出和恢复语义。

SSE 支持实时事件流、恢复语义和流式 token 展示。

推荐操作顺序

如果你第一次打开这个页面，建议不要从底层 JSON 开始。先走下面这条最短路径，能最快确认系统是活的、Agent 是通的、结果是看得见的。

1
检查服务状态
先确认后端可访问。状态正常后，再做后续演示才有意义。
2
打开 Agent 列表
先选一个 agent。最稳定的是 echo-agent-1；想直接验证通用文本、分析、翻译、推理链路时优先选 deepseek-chat-agent。
3
优先体验自然语言工作流
它已经把用户输入、workflow 组装、流式输出、最终结果都封装好了，是当前最像真实产品入口的一页。
4
最后回到执行跟踪
在那里看 execution 核心状态、step 卡片、时间线和原始事件，确认统一 execution 模型是否足够清晰。

这几个页面分别看什么

创建任务
最适合验证单次 agent 执行是否正常，尤其是 echo 或单次 LLM 请求。
自然语言工作流
现在它既是用户输入入口，也是 workflow 节点编排入口。适合直接演示“用户输入 -> 多节点处理 -> AI 输出”。
执行跟踪
最适合调试、复盘、对照 execution 视图和 SSE 事件是否足够表达业务状态。
Agent 列表
最适合先确认当前服务里到底有哪些 agent 真注册了，以及它们各自支持什么能力。

两个必须理解的概念

Execution 是统一视角
不管你是创建 task 还是 workflow，最后都回到 execution 去查状态、结果、步骤和事件。
Recovery 是恢复语义，不是主状态
recovery_status 只说明恢复链路，比如 interrupted、resumed、superseded，它不替代 status。
流式输出看两个区域
正文持续增长看“正文流式输出”，事件变化看“过程事件流”。这两者都是可观测过程，不代表模型内部隐藏推理。
模板必须匹配 agent
echo-agent-* 用 message/text 风格；DeepSeek/OpenAI 要用 prompt/system_prompt 风格，否则会直接报参数错误。

建议你先做哪三件事

路径 A：最快看到结果
检查服务状态 → 进入“自然语言工作流” → 选 DeepSeek → 点击创建 → 在当前页直接看流式正文和最终结果。
路径 B：验证统一 execution
创建一个 task，再创建一个自然语言工作流，最后去“执行跟踪”对比二者在统一 execution 面板里的差异。
路径 C：验证恢复与事件
执行 workflow 后重点看事件时间线、恢复字段和 SSE 更新，确认前端是否能消费恢复链路。

创建工作流

这里走 POST /api/v1/workflows/execute。页面会根据你选择的 agent 自动切换 workflow 模板：echo 用回显模板，DeepSeek/OpenAI 用 prompt 风格模板。

工作流页说明

如果你想看 step 状态、summary 聚合、恢复字段，这一页比任务页更有代表性。

默认结构
两个步骤：先 echo，再把上一步结果传给第二步处理。
推荐观察点
创建后去“执行跟踪”看 summary、steps、时间线是否清晰。

工作流模版

这里对应 Coze 风格的工作流模版入口。你不需要手改底层 JSON，可以直接添加多个节点，定义每个节点的 agent、处理模式、提示词和是否基于上一步结果继续处理。

这一页做了什么

它会把你输入的自然语言内容写进 workflow step 的 prompt，然后把最终结果统一展示到 execution 视图里。

节点能力
你现在可以自定义添加多个节点。每个节点都可以单独设置 agent、action、prompt、system_prompt。
节点依赖规则
默认按顺序串联。第一个节点直接处理用户输入，后续节点可选择是否基于上一步结果继续处理。
结果位置
现在除了“执行跟踪”页，这一页右侧也会直接显示当前自然语言工作流的输出结果。

如何选择 Agent

如果你只是想理解这个前端，先不要追求最复杂的模型。优先选稳定、一定存在的 agent 来验证主链路。

推荐优先级
先用 echo-agent-1，确认接口、执行视图和时间线都正常，再试其他 agent。
按钮作用
用于任务 和 用于工作流 不只是替换 agent 名字，还会按当前 agent 类型切换整套模板。
模板规则
echo-agent-* 使用 message/text 模板；deepseek-chat-agent 和 openai-gpt-agent 使用 prompt/system_prompt 模板。
当前最推荐的演示路径：先检查服务状态，再进入“自然语言工作流”直接创建一次带流式输出的 workflow；确认结果正常后，再回到“执行跟踪”看 execution、steps、timeline 和 recovery 字段。这样既能看产品效果，也能看底层执行模型。
